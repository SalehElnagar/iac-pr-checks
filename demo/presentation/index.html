Audience takeaway

You should leave with a simple mental model: three required PR checks, changed-only targeting, and a PR comment that tells reviewers exactly what failed.

The Problem

😰 Without Automation

  • Reviewers burn time on whitespace and obvious mistakes
  • Invalid Terraform or Terragrunt changes reach main
  • Lint issues show up late or not at all
  • PR feedback is inconsistent across engineers
  • Merges depend on memory instead of enforceable checks

✅ With IaC PR Checks

  • Every PR gets fmt, validate, and lint
  • Checks run only against impacted modules and Terragrunt units
  • Branch protection blocks merge until the PR is green
  • A single PR comment shows results, scope, and issues
  • Humans can focus on design and risk, not avoidable noise

Smart Target Discovery

  • Classifies: Module? Terragrunt unit? Shared HCL?
  • Resolves module → live unit dependencies
  • Generates a runtime manifest for the three jobs

Stage 1

Formatting Checks

What It Does

  • terraform fmt -check -diff -recursive on impacted module roots
  • terragrunt hcl fmt --check --diff on impacted HCL files
  • Fails fast without rewriting files in CI
  • Writes a JSON report for the PR summary

Report Capture

Results written to fmt-result.json, uploaded as a GitHub Actions artifact. The lint job later downloads all reports and composes the unified PR comment. 

#!/usr/bin/env bash
# run_iac_fmt.sh (simplified)

discover_targets "${json_path}"

# ── Terraform ──
for dir in $(terraform_fmt_dirs); do
  terraform -chdir="${dir}" \
    fmt -check -diff -recursive
done

# ── Terragrunt ──
for file in $(terragrunt_fmt_files); do
  terragrunt hcl fmt \
    --check --diff --file "${file}"
done

write_fmt_report "${result}"
Stage 2

Validation Checks

What It Does

  • terraform init -backend=false -input=false + validate
  • terragrunt hcl validate --inputs per impacted unit
  • Catches missing inputs, bad refs, and type problems
  • Runs in parallel with formatting for fast feedback

Why It Stays CI-Safe

No plan, no apply, no cloud credentials. The helper rewrites GitHub SSH module sources to HTTPS so public tagged modules can be fetched safely during init. 

#!/usr/bin/env bash
# run_iac_validate.sh (simplified)

discover_targets "${json_path}"

# ── Terraform ──
for dir in $(terraform_validate_dirs); do
  run_with_public_github_module_access \
  terraform -chdir="${dir}" \
    init -backend=false -input=false
  run_with_public_github_module_access \
  terraform -chdir="${dir}" validate
done

# ── Terragrunt ──
for dir in $(terragrunt_validate_dirs); do
  terragrunt hcl validate \
    --inputs --working-dir "${dir}"
done

write_validate_report "${result}"
Stage 3

Lint & Report

TFLint Checks

  • tflint --chdir=<module> per Terraform dir
  • Uses repo-level .tflint.hcl config
  • --call-module-type=none for offline safety
  • Catches deprecated syntax & best practices

📋 Unified PR Comment

  • Downloads fmt & validate report artifacts
  • render_pr_comment.py composes Markdown
  • upsert_pr_comment.py updates existing comment
  • Shows result table, issues, scope, and notes

Why You Can Turn This On

☁️

No Cloud Access Required

These checks stop at fmt, validate, and lint. No plan, no apply, and no dependency on real backend access.

🔑

Minimal GitHub Permissions

Most jobs only need repository read access. Only the PR comment step needs write permission, and checkout does not persist credentials.

📌

Predictable Tooling

Terraform, Terragrunt, and TFLint are pinned to known versions, and the downloaded binaries are checksum-verified before use.

🚦

Safe To Enforce

The three job names stay visible on PRs and on main, so branch protection can reliably block merges until the checks pass.

Key Takeaways

The goal is not more CI. The goal is better infrastructure pull requests.

1

Automate The Obvious

Formatting, configuration validation, and linting should fail before a human reviewer has to point them out.

2

Keep Checks Scoped

Target only the impacted Terraform modules and Terragrunt units so the workflow stays fast enough for normal PR work.

3

Make Results Actionable

Use clear check names, one clean PR comment, and required status checks so teams know what failed and cannot ignore it.

When automation handles the avoidable issues, reviewers can spend their time on architecture, risk, and logic.
